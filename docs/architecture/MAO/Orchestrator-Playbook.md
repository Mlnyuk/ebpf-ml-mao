# 오케스트레이터 운영 규칙

이 문서는 현재 저장소에서 메인 오케스트레이터가 어떤 기준으로 작업을 분해하고, 어떤 순서로 에이전트를 호출하고, 어떤 조건에서 최종 반영을 승인하는지 정리한 운영 규칙서다.

## 1. 오케스트레이터의 역할

오케스트레이터는 아래 4개 책임을 가진다.

- `Planner`
  - 목표를 단계와 작업 단위로 분해한다.
- `Router`
  - 각 작업을 가장 적합한 에이전트에 할당한다.
- `Integrator`
  - 산출물을 병합하고 충돌을 조정한다.
- `Judge`
  - 테스트, 문서, 구현 정합성을 확인하고 최종 승인 여부를 결정한다.

현재 이 저장소에서는 메인 실행자가 위 4개 역할을 모두 겸한다.

## 2. 기본 운영 원칙

- 한 번에 하나의 단계만 명확히 진행한다.
- 각 단계는 완료 기준이 있어야 한다.
- 설계와 구현은 분리하되, 문서와 코드의 경로는 함께 갱신한다.
- 최종 반영 전에는 항상 테스트 또는 최소 검증 명령을 실행한다.
- 다중 에이전트 결과는 그대로 믿지 않고 메인 오케스트레이터가 최종 판단한다.

## 3. 에이전트 배정 원칙

### Claude Code

적합한 작업:

- 요구사항 구조화
- 아키텍처 문서 초안
- 설계 대안 비교
- 인터페이스 정의 초안

### Codex

적합한 작업:

- 실제 코드 수정
- 테스트 작성
- 디렉터리 구조 정리
- 구현 결과 통합

기본 역할:

- 메인 실행자
- 최종 반영자

### Gemini CLI

적합한 작업:

- 대규모 문서/코드 탐색
- 비교 분석
- 로그/설정 차이 확인
- 구현 리스크 요약

주의:

- 응답 시간이 길 수 있으므로 blocking task로 두지 않는다.
- 결과는 참고 근거로만 쓰고 최종 결론은 메인 실행자가 낸다.

## 4. 작업 분해 규칙

모든 구현 작업은 아래 5개 항목으로 정리해서 시작한다.

- `goal`
- `scope`
- `non_goals`
- `deliverables`
- `verification`

## 5. 승인 기준

오케스트레이터는 아래를 만족할 때만 단계를 완료로 본다.

- 코드가 저장소 구조와 일치한다.
- 문서의 실행 예시가 실제 경로와 맞다.
- 테스트 또는 최소 검증 명령이 통과한다.
- README 또는 단계 문서에 변경 사항이 반영된다.

## 6. 저장소 운영 규칙

- 새로운 단계는 `docs/steps/stepN/` 아래에 기록한다.
- 새로운 샘플은 `samples/stepN/` 또는 `samples/phaseN/` 아래에 둔다.
- 배포 관련 파일은 `deploy/` 아래에 둔다.
- 설계 문서는 `docs/architecture/` 아래에 둔다.
- 환경/운영 메모는 `docs/specs/` 아래에 둔다.

## 7. 추천 실행 순서

실제 작업은 보통 아래 순서로 진행한다.

1. 목표와 범위를 고정한다.
2. Claude 또는 Gemini에 설계/탐색 작업을 보낸다.
3. 메인 실행자가 코드를 수정한다.
4. 테스트와 경로 점검을 한다.
5. 단계 문서와 README를 갱신한다.
6. Git 상태 확인 후 커밋/푸시한다.
