# eBPF 기반 Kubernetes 환경 관리 구현 방법론

이 문서는 `eBPF를 활용한 Kubernetes 환경 관리`를 어떤 방식으로 구현할 수 있는지에 대한 현실적인 접근 방법 3가지를 정리한다. 목표는 단순 모니터링이 아니라, 클러스터 내부 이벤트를 관찰하고, 정책과 연결하고, 운영 자동화까지 확장 가능한 구조를 만드는 것이다.

## 방법론 1. 관측 중심 DaemonSet 방식

가장 빠르게 시작할 수 있는 방식은 각 노드에 `DaemonSet` 형태의 에이전트를 배포하고, 이 에이전트가 eBPF 프로그램을 로드해 네트워크, 프로세스, 시스템 콜, 컨테이너 이벤트를 수집하는 구조다.

### 핵심 아이디어

- 각 노드에 하나의 eBPF agent를 배포한다.
- agent는 `tracepoint`, `kprobe`, `cgroup hook`, `tc/XDP` 등을 사용해 이벤트를 수집한다.
- 수집한 이벤트를 `ring buffer` 또는 `perf buffer`로 userspace에 전달한다.
- userspace collector가 이를 정규화해 중앙 수집기로 보낸다.

### 장점

- 구현 난도가 상대적으로 낮다.
- 기존 k8s 운영 환경에 점진적으로 도입하기 쉽다.
- `Cilium`, `Tetragon` 같은 오픈소스의 구조를 참고하기 좋다.

### 단점

- 초기에 "보는 것"은 쉽지만 "제어하는 것"까지 확장할 때 아키텍처 재정비가 필요하다.
- 이벤트 volume이 커지면 중앙 수집기와 저장소 설계가 병목이 될 수 있다.

### 적합한 경우

- 먼저 가시성 확보가 필요한 경우
- Pod 간 네트워크 흐름, 프로세스 실행, 비정상 syscall 탐지부터 시작하려는 경우
- PoC와 운영 초기 버전을 빠르게 만들고 싶은 경우

## 방법론 2. 정책 중심 Controller + CRD 방식

두 번째 방식은 Kubernetes 네이티브 모델을 중심에 두고, eBPF는 정책 실행 엔진으로 사용하는 방식이다. 사용자는 `CRD`로 정책을 선언하고, 컨트롤러가 이를 해석해 각 노드의 eBPF agent에 반영한다.

### 핵심 아이디어

- `TracingPolicy`, `NetworkPolicyExtension`, `RuntimeGuard` 같은 CRD를 정의한다.
- 중앙 controller가 CRD 변경을 watch한다.
- controller가 정책을 노드 단위 agent 설정으로 변환한다.
- 각 agent는 필요한 eBPF 프로그램을 attach하거나 설정을 갱신한다.

### 장점

- Kubernetes 운영 방식과 잘 맞는다.
- 정책, 상태, 리컨실리에이션이 명확해져 운영 자동화에 유리하다.
- 장기적으로 RBAC, 멀티 테넌시, 감사 이력 관리에 강하다.

### 단점

- 초기 설계 비용이 크다.
- CRD 스키마와 controller 책임 분리를 잘못하면 복잡도가 급격히 증가한다.
- eBPF 자체보다 control plane 설계가 더 어려워질 수 있다.

### 적합한 경우

- 단순 수집이 아니라 정책 기반 운영 플랫폼을 만들려는 경우
- 팀이 Kubernetes operator/controller 패턴에 익숙한 경우
- 향후 UI, 알림, 자동 조치까지 포함한 제품형 구조를 원하는 경우

## 방법론 3. 서비스 메시/보안 플랫폼 연계 확장 방식

세 번째 방식은 eBPF만으로 모든 것을 새로 만들기보다, 기존 네트워크 또는 보안 플랫폼과 연계하는 방식이다. 예를 들어 `Cilium`을 네트워크 데이터 plane으로 사용하고, 상위 계층에서 운영 관리 기능을 덧붙이는 접근이다.

### 핵심 아이디어

- 저수준 packet 처리와 일부 policy enforcement는 기존 eBPF 기반 플랫폼에 맡긴다.
- 별도 control layer에서 운영 정책, 이벤트 분석, 위험 탐지, 자동 대응을 구현한다.
- 필요한 부분만 커스텀 eBPF probe로 보강한다.

### 장점

- 커널 호환성, verifier 대응, 네트워크 data plane 구현 부담을 크게 줄일 수 있다.
- 운영 안정성을 빠르게 확보할 수 있다.
- 차별화 포인트를 운영 관리 레이어에 집중할 수 있다.

### 단점

- 외부 플랫폼 구조에 종속된다.
- 내부 동작을 완전히 제어하기 어렵다.
- 제품 차별성이 약해질 수 있어, 어떤 부분을 직접 구현할지 경계 설정이 중요하다.

### 적합한 경우

- 상용화 또는 데모 속도가 중요한 경우
- 네트워크/보안 기능보다 "운영 관리 플랫폼"이 핵심 목표인 경우
- 팀이 eBPF 저수준 개발보다 k8s control plane과 플랫폼 개발에 더 강한 경우

## 추천 방향

현실적으로는 `방법론 1`로 시작해서, 운영 모델이 정리되면 `방법론 2`로 확장하는 단계형 접근이 가장 안전하다.

추천 순서는 다음과 같다.

1. DaemonSet agent로 이벤트 수집 PoC를 만든다.
2. 어떤 이벤트를 정책 자원으로 추상화할지 정의한다.
3. CRD와 controller를 도입해 정책 중심 구조로 옮긴다.
4. 필요하면 Cilium 같은 기존 eBPF 플랫폼과 연계해 데이터 plane 복잡도를 줄인다.

즉, 초기에는 `관측`, 중기에는 `정책`, 장기에는 `플랫폼 통합`으로 발전시키는 전략이 가장 구현 가능성이 높다.
