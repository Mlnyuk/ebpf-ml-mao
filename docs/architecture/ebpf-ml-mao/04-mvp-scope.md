# 4. MVP Scope

MVP의 목적은 `실제 클러스터 또는 테스트 환경에서 eBPF 이벤트를 수집하고, ML 기반 이상 탐지를 수행한 뒤, 멀티에이전트 협업 결과를 리포트로 내는 것`이다.

## MVP에 반드시 포함할 것

### 1. 입력 소스

- Tetragon 이벤트 1종 이상
- Prometheus 메트릭 1종 이상
- 테스트용 정상/이상 샘플 데이터셋

최소 권장:
- `process_exec`
- `network connection`
- CPU 또는 메모리 사용량

### 2. 내부 파이프라인

- 이벤트 정규화
- 시간 윈도우 기반 feature 추출
- 기본 anomaly scoring
- 멀티에이전트 분석 파이프라인
- 최종 JSON/Markdown 리포트 생성

### 3. ML 범위

MVP에서는 복잡한 온라인 학습보다 추론 중심으로 간다.

권장:
- Isolation Forest
- One-Class SVM
- Autoencoder 중 1개

현실적인 선택:
- 첫 버전은 `offline trained model + online inference`

### 4. 에이전트 범위

반드시 4개 역할을 모두 분리할 필요는 있지만, 구현 복잡도를 줄이기 위해 내부적으로는 2단계부터 시작할 수 있다.

초기 구현 순서:
1. `Summarizer`
2. `Analyst`
3. `Correlator`
4. `Reviewer`

단, 코드 인터페이스는 처음부터 4역할 확장을 전제로 설계한다.

## MVP에서 제외할 것

- 자동 복구
- Kubernetes 리소스 수정
- `kubectl` 실행기
- RBAC가 필요한 조치 자동화
- 사람 승인 워크플로우
- 장기 상태 저장소
- 실시간 대시보드

이것들은 프로젝트를 무겁게 만들고, 현재 목표인 `탐지 및 분석 오케스트레이션`을 흐린다.

## 데모 시나리오

MVP 데모는 아래 셋 중 하나만 성공해도 충분하다.

### 시나리오 A. 비정상 프로세스 실행

- 특정 워크로드에서 평소와 다른 프로세스 실행 발생
- eBPF/Tetragon이 이벤트 수집
- ML 점수 상승
- 에이전트가 이상 실행 시퀀스라고 보고

### 시나리오 B. 네트워크 행위 이상

- 평소 없는 외부 연결 또는 burst 발생
- 메트릭과 이벤트를 묶어 상관분석
- 최종 리포트에서 의심 네트워크 이상으로 표시

### 시나리오 C. 리소스 급변 + 행위 조합

- CPU 급증과 프로세스 생성 이벤트가 동시 발생
- 단일 메트릭이 아니라 행위 조합 이상으로 해석

## 구현 우선순위

1. 이벤트 수집과 내부 스키마 확정
2. feature 추출기 작성
3. ML 추론 연결
4. Summarizer/Analyst 연결
5. Correlator/Reviewer 연결
6. 리포트 포맷 마감

## 완료 기준

아래를 만족하면 MVP 완료로 본다.

- 샘플 또는 실제 이벤트 입력이 들어온다.
- 모델이 anomaly score를 낸다.
- 최소 2개 이상의 에이전트가 결과 생성에 참여한다.
- 최종 리포트가 사람이 읽을 수 있는 형태로 출력된다.
- 정상 케이스와 이상 케이스를 구분하는 데모가 가능하다.
